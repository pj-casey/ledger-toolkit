Data Processing

Ingestion & Parsing

Accepts structured JSON log exports, plain text, and mixed-format files via drag-and-drop or file picker. Input extensions: .json .txt .log

Three-tier parser: The parser attempts structured parsing first, falling back to a recovery parser for malformed JSON, and finally a brace scanner that extracts parseable JSON objects from corrupted or partially-truncated files. Handles UTF-8 BOM and normalizes top-level arrays vs. objects.

Mobile log synthesis: When exportLogsMeta is absent (mobile log format), the parser reconstructs metadata — platform, app version, and account IDs — from bridge schedule messages embedded in the event stream.

Post-parse normalization: All entries are chronologically sorted and assigned a stable logIndex used as the universal cross-reference key throughout the tool. Reverse-chronological mobile exports are detected and reversed before indexing.

Tested with logs containing 15,000+ entries. All parsing runs client-side in the browser — no data is transmitted to any server during parsing.

Data Quality Assessment

Automated quality scoring on a 0–100 scale evaluates log completeness across four independent dimensions:

  • Metadata presence — device model, firmware version, OS, user ID, app version
  • Account data — sync durations, reported balances, operation counts
  • Error context — stack traces, payload data, query arguments attached to errors
  • Session integrity — timestamp consistency, entry count, event type diversity

Grade thresholds: Excellent (85+), Good (65+), Fair (40+), Poor (<40). Each dimension contributes a weighted sub-score; the breakdown is available for granular assessment.

Mobile log normalization adjusts expectations for platform-specific data gaps — mobile logs are not penalized for absent sync data that is structurally unavailable on that platform.

Quality scoring enables objective comparison between log exports — a 90-score log from a 2-minute session provides more investigative surface than a 45-score log from a 30-minute session.

Device & Hardware Analysis

Device Identification

Multi-source identification with automatic fallback chain. The extractor attempts each method in order, stopping when both modelId and fw are resolved:

  1. Direct metadata fields — d.modelId, d.modelName, d.deviceVersion from exportLogsMeta
  2. Target ID bitmask analysis — applies (targetId & 0xFFFF0000) >>> 0 against 6 known device masks. Same algorithm as @ledgerhq/devices. Scans data.targetId, data.data.target_id, and Manager API URL params. Primary identification method for Ledger Wallet 4.0+ logs.
  3. Manager API response — reads get_device_version response data.data.name (e.g., "Nano S Plus")
  4. Firmware path prefix — parses install paths such as nanos+/1.5.1/... to extract model and firmware. Covers apex_p / apex for Gen5 devices.
  5. Device apps result backfill — extracts model from actions-manager-event payload when earlier methods are inconclusive

Supported devices: Nano S, Nano X, Nano S Plus, Stax, Flex (europa), Nano Gen5 (apex). Firmware version extraction includes update availability detection when the current firmware can be compared against the latest known release.

Software version-aware branding: Detects whether the session used "Ledger Live" (v2.x) or "Ledger Wallet" (v4.x) via userAgent parsing and version number analysis. All diagnostic text adjusts accordingly.

Application Protocol Data Unit (APDU) Analysis

Extracts device communication commands from log entries. Each APDU record includes command direction (sent to device / received from device), raw instruction byte, and status code.

Status code interpretation:

  • 9000 — Success
  • 6985 — Conditions not satisfied (user rejected on device)
  • 6982 — Security status not satisfied (device locked or PIN not entered)
  • Additional codes are parsed from the error knowledge base and cross-referenced against known patterns

TLV parsing: GetAppAndVersion responses (instruction B001) are parsed as Tag-Length-Value structures to identify which applications were open on the device during the session.

Rejection counting and summary generation per session. Supports both the legacy APDU format and the LLv4 Device Management Kit (DMK) logger format used in Ledger Wallet 4.0+.

Installed Application Detection

Three detection methods with priority fallback:

  1. Manager result — full installed app list with versions and update status from actions-manager-event responses. Source label: manager-result.
  2. APDU TLV parsing — GetAppAndVersion exchanges identify the running application during the session. Source label: apdu-only. Single app only.
  3. Catalog cross-reference — hw install events matched against apps-by-target API catalog data. Source label: catalog-only. Covers LLv4 sessions where manager result is unavailable.

Required app inference: Account currencies are mapped to the device applications they depend on (via a 46-entry CURRENCY_TO_APP map). Each inferred required app is compared against the detected installed set to produce status classifications: installed + current, installed + outdated, or not installed.

Chain enrichment: When 2 or more accounts require the same device app, the app row lists all dependent chain tickers. Missing or outdated apps are surfaced as visual indicators on account cards.

Session Reconstruction

Temporal Event Classification

Every log entry is classified by type: action, analytics, bridge, countervalues, error, network, persistence, walletsync, live-dmk-logger, and others. Each type is assigned a consistent color used identically across all visualizations — session strip bars, row type badges, legend chips, Issues strip, and ErrCard breadcrumbs.

Per-entry metadata extraction: timestamp (normalized from date or timestamp field), type, log level, message string, structured data payload, query arguments, payload fields, and embedded error objects.

Session boundary detection: first and last timestamps establish total session duration. Entry count and type diversity contribute to data quality scoring.

Session Visualization

The session is divided into 80 equal time buckets. Each bucket renders as a stacked bar where bar height represents event density (total entry count) and color segments represent event type distribution, using the same canonical color map as row badges.

Legend chips: One chip per event type present in the session, showing type name, color, and entry count. Hovering a chip dims all non-matching bar segments to 15% opacity. Clicking a chip filters the entry list to that type. State is shared with the type dropdown filter — changes in either update both.

Error dot baseline: Error entries are rendered as individual colored dots on a dedicated baseline below the bars. Each dot is a clickable navigation target — click scrolls to and expands that entry in the list.

Bucket hover tooltip: Hovering any bar shows timestamp, total event count for that bucket, per-type breakdown, and sample message. Clicking the bar scrolls to and expands the nearest log entry to the bucket's midpoint.

Event Grouping

Consecutive same-type events within a 1.5-second temporal window are automatically collapsed into expandable group headers. Group headers display: event type, repeat count, time range, and total duration.

Minimum group size: 3 events — sequences smaller than this remain as individual rows. Error entries are unconditionally excluded from grouping and always appear as individual rows regardless of proximity or type match.

Groups are keyed by the logIndex of the first entry in the group, allowing stable expand/collapse state across filter changes. Toggleable between Grouped (default) and Flat mode via a header button.

Typical compression ratio: 50–70% reduction in visible rows on standard production logs (e.g., 2,500 raw entries → approximately 80 visible rows after grouping).

Account-Correlated Filtering

Filters the full event stream to entries that reference a specific account. The matcher performs case-insensitive substring search across five fields per entry: message string, type string, serialized data payload, serialized query arguments, and serialized payload fields.

Match criteria (OR logic across fields, AND with other filters):

  • Address — substring match when address length ≥ 8 characters
  • Currency ID — exact chain currency identifier (e.g., ethereum, solana)
  • Ticker symbol — 3–5 character ticker (e.g., ETH, SOL)
  • Chain name — human-readable chain name

Activated from the Accounts section via Shift+click on a health tile or the Timeline button on account cards. A filter banner in the Timeline header displays the active account and provides one-click clear. Combines with type filter and text search as AND logic.

Error Analysis & Diagnosis

Pattern-Based Error Classification

82-pattern error knowledge base. Each pattern specifies a matching strategy (regex or substring), category (Hardware / Software / Server), severity (Critical / Warning / Info), diagnostic title, recommended agent action, and list of common causes.

Priority matching: When multiple patterns match a single error entry, the highest severity match is selected. Within the same severity, the match with the longer match string wins (specificity tiebreaker).

APDU subcode extraction: When a generic transport error is detected, the system scans the error data and message for embedded hexadecimal status codes and attempts a secondary pattern match against the more specific code. This converts generic "device communication failed" errors into actionable diagnoses when the specific rejection code is present.

Unmatched errors are classified as "Unexpected error" (Info severity) with full raw data preserved for manual inspection via the Raw JSON viewer.

ERR_DB text localization: All diagnostic advice text undergoes runtime substitution — "Ledger Live" is replaced with "Ledger Wallet" when the session brand is detected as the newer product, without modifying the knowledge base array itself.

Temporal Error Analysis

Error clustering analysis computes the time span between the first and last error relative to total session duration. Automatic plain-English interpretation is generated:

  • Single error — position reported as a percentage into session (e.g., "at 73% of session")
  • All errors within 2 seconds — "All N errors within Xs — likely a single cascade event"
  • Errors within 30% of session span — "likely related"
  • Errors spread beyond 30% — "likely independent issues"

Repeat detection: Errors sharing the same diagnostic title are counted and grouped. When a pattern appears 2 or more times, it is surfaced as an interactive badge in the Issues header (e.g., "↻ Price data unavailable ×4"). Clicking a badge applies a pattern filter to the error list.

Error Context Breadcrumbs

For each error entry, the 5 immediately preceding log entries are locatable by scanning backward through the full entry array from the error's logIndex. These entries are renderable as a breadcrumb trail in the error detail panel.

Each breadcrumb shows: relative time offset from the error (e.g., −0.3s), event type with a TC-colored badge matching the type badge system used throughout the tool, and a truncated message string. The final row in the trail is the error itself, highlighted in the error's severity color.

One-click navigation from any error detail panel to the error's position in the full Timeline — this navigation clears all active Timeline filters and grouping to guarantee the target entry is visible, then scrolls to it.

Error-to-Account Correlation

Automatic scanning of error messages, structured data payloads, and raw payload fields for strings matching known account addresses. When a match is found (candidate string ≥ 8 characters), the error is linked to the corresponding account.

Linked accounts appear as interactive navigation chips on error detail cards. Clicking an account chip navigates to that account in the Accounts section. The correlation map is computed once at log load time using a Map<logIndex, account> structure keyed on entry position.

The reverse pathway also exists: clicking an error badge on an account card in the Accounts section navigates to the Issues section with a filter applied to show only errors associated with that account's address.

Interactive Severity & Category Filtering

Severity filter chips (Critical / Warning / Info) and category filter chips (Hardware / Software / Server) are rendered in the Issues section header. Clicking a chip applies that filter. Filters combine with AND logic — activating Critical + Hardware returns only errors that are simultaneously critical severity and hardware category.

Hover interaction: Hovering a severity chip dims non-matching error indicators on the session strip in real time, without applying a persistent filter. Hovering a category chip produces the same effect for the category dimension. This enables visual inspection of distribution before committing to a filter.

Error session strip: Stacked bars using the same color and bucket system as the Timeline strip. Error entry segments are rendered at full opacity; non-error context segments are rendered at 25% opacity. Hovering any bar renders a tooltip showing the timestamp, error count, total event count, and error details for that bucket. Clicking selects the nearest error.

Account & Asset Analysis

Account Extraction

Primary extraction from SyncSuccess analytics events (Ledger Live / Ledger Wallet 2.x–3.x format). Each event carries per-account sync duration, operation count, and reported balance. Secondary extraction from bridge sync sessions (Ledger Wallet 4.0 / LLv4 format).

LLv4 bridge sync fallback: When no SyncSuccess analytics events are present, the extractor scans for type: "bridge" entries containing "SyncSession finished". When found, the total session duration is applied as a sync duration across all accounts, preventing false "no sync data" alerts on newer log formats.

Per-account data extracted: chain identification, address, derivation mode, operation count, log-reported balance, spendable balance, sync duration, token list with contract addresses and operation counts. Account IDs extracted from exportLogsMeta.accountsIds for cross-referencing with external systems.

Address Derivation & Conversion

Three chains store non-explorer-compatible credential formats in logs. The tool automatically converts these to usable addresses:

  • Cardano — hex credential bytes → bech32 stake address (stake1...). Conversion via cardanoCredToStake().
  • TON — 32-byte hash → user-friendly bounceable address (UQ...) with CRC16 checksum. Conversion via tonHexToAddr().
  • Stacks — compressed public key → c32check mainnet address (SP...). Conversion via stacksPubkeyToAddr().

Derived addresses are surfaced with copy buttons and direct block explorer links. All three conversions run entirely client-side with no external API calls.

Derived addresses are computed from log data — always verify with the customer before taking action based on derived values.

UTXO Extended Public Key Analysis

Full BIP32 hierarchical deterministic key derivation for Bitcoin, Litecoin, Dogecoin, and Bitcoin Cash. Derives individual addresses from xpub, ypub, and zpub keys along receive (m/0/i) and change (m/1/i) chains.

Supported derivation modes:

  • BIP84 — Native SegWit (zpub, P2WPKH, bc1q... addresses)
  • BIP49 — SegWit (ypub, P2SH-P2WPKH, 3... addresses)
  • BIP44 — Legacy (xpub, P2PKH, 1... addresses)

Per-address live balance query and transaction count via public blockchain APIs. Gap limit detection: scanning stops after consecutive unused addresses are found (gap limit = 20 for receive chain, configurable). Rate limiting handled with automatic retry up to 5 times with exponential backoff.

Per-address explorer links for independent on-chain verification. Uses bitcoinjs-lib 5.2.0 for address derivation, running entirely in the browser via bundled library.

EVM Address Grouping

Accounts sharing the same Ethereum-format address (hex strings ≥ 20 characters) are automatically identified as an EVM group. This is common when a customer uses Ethereum alongside L2 and EVM-compatible chains (Polygon, Arbitrum, Optimism, BSC, Base, etc.) that share the same address space.

Group rendering: shared address header with copy button, chain tile count, and combined fiat value aggregated across all accounts in the group. The Ethereum account sorts first within each group; remaining chains sort by fiat value descending.

Grouping is applied after the full account list is rendered — individual account cards within a group retain their full detail, error badges, and xpub scan buttons.

Account Health Visualization

Per-account health tile with chain-colored background derived from the chain registry color and a 3-letter ticker label. Tile opacity reflects account status: funded (full), empty (reduced), no sync data (amber border).

Corner indicator system:

  • Red dot (top-right) — one or more errors are correlated to this account
  • Purple dot (bottom-right) — account has an xpub / is scannable via the UTXO scanner
  • ? label — chain is not supported for live balance fetching

Hover popover: chain name, truncated address, live balance and fiat value, sync status (no sync / loading / empty), derivation mode, error count, and xpub/explorer availability flags. Popover is pointer-events-none and clamped to container width.

Portfolio overlay: Activated via "Portfolio ▸" button. Tiles are resized proportionally to fiat value (36–80px) and sorted by value descending. Clicking any tile applies a chain filter and closes the overlay.

On-Chain Verification

Live Blockchain Balance Fetching

Automatic balance retrieval for 40+ blockchain networks from public APIs. No API keys, authentication, or server-side components required. All requests originate from the user's browser.

Coverage by tier:

  • EVM chains (23 networks) — Ethereum, Polygon, BSC, Arbitrum, Optimism, Avalanche, Base, Fantom, Celo, and others via JSON-RPC eth_getBalance against publicnode.com endpoints
  • UTXO chains — Bitcoin (mempool.space), Litecoin/Dogecoin (BlockCypher), Bitcoin Cash (Blockchair)
  • Major L1s — Solana, Tezos, Cosmos, XRP, Polkadot, Cardano, NEAR, Tron, TON, Stellar, Hedera, Algorand, and others via chain-specific REST/RPC APIs

Fiat pricing: CoinGecko API with chain currency IDs mapped via a 40+ entry COINGECKO_IDS constant. USD prices batch-fetched after balances complete.

Request staggering at 200ms intervals to avoid rate limiting. Per-account status tracking: loading, ok, error, unsupported. Portfolio aggregation: sum of all accounts' USD values displayed as a stat card on the Overview. Accounts with unsupported status are identified with a ? indicator on health tiles.

Token Resolution

Token contract address resolution for 45+ EVM and non-EVM chains via a TOKEN_CONTRACTS registry and per-chain URL builders (TOKEN_URLS). When an exact contract resolution is available, a direct token explorer link is generated. When it is not, a TOKEN_SEARCH fallback generates an explorer search URL for 8 supported chains.

Token badges on account cards are rendered as clickable links. Token operation counts from log sync data are preserved per token per account. EVM chain IDs are mapped via a 22-entry EVM_CHAIN_IDS constant used to construct the correct explorer URL for each network.

Cross-Domain Correlation

Connected Navigation

The following data correlation pathways are implemented as direct navigation actions:

  • Error → Account — address matching in error data links errors to specific accounts; linked account chips navigate to the Accounts section
  • Error → Timeline — "View in Timeline" clears all active filters, disables grouping, expands the target entry, and scrolls to it
  • Account → Issues — error badge on account cards applies an address filter to the Issues section, showing only errors referencing that account
  • Account → Timeline — Shift+click on a health tile or the Timeline button on account cards applies account-correlated filtering to the full event stream
  • Timeline → Entry detail — clicking any point on the session visualization strip navigates to the corresponding raw event
  • Overview → All sections — stat cards display entry, account, error, and portfolio counts that establish baseline expectations before investigation

Multi-Source Data Triangulation

Three independent balance data sources are available simultaneously:

  • Log-reported balance — from sync events captured in the log at the time of the session
  • Live blockchain balance — fetched from public APIs at the time of analysis
  • App.json balance — from the application's local state file, when loaded in Customer View

Comparison across these sources enables detection of:

  • Sync failures — log balance differs from live balance
  • Stale local cache — app.json balance differs from live balance
  • Missing accounts — accounts present in app.json but absent from the log
  • Unreported assets — live balance is positive but no sync data was captured

Network & Protocol Analysis

HTTP Request Analysis

Extraction and display of all HTTP requests captured in the log. Each request record includes URL, HTTP method, status code, and associated error data when present.

Status code classification: success (2xx), client error (4xx), server error (5xx). Error summarization at the section header level: total error count with server vs. client error breakdown. This enables rapid determination of whether failures were caused by server-side problems or client configuration issues.

Request URLs are displayed in full and are useful for identifying which Ledger or third-party APIs were contacted during the session, including firmware update endpoints, Manager API calls, and pricing service requests.

Session Activity Detection

Automatic detection of significant user activities during the session based on analytics event and log entry patterns:

  • Manager access — device app management session detected from manager-related events
  • Swap activity — swap browsing and execution events
  • Device scanning — transport scanning events indicating device connection attempts

Analytics event parsing: App Start metadata captures user ID, OS, platform, device model, and app version at session open. Portfolio analytics events record portfolio metrics at session time. SyncSuccess events capture per-account sync outcomes including operation count and balance.

Software version-aware branding: The tool detects whether the session originated from "Ledger Live" (versions 2.46–~2.131) or "Ledger Wallet" (versions ~2.132+, 4.0+). Detection uses userAgent string parsing from exportLogsMeta with version number fallback. All labels, error advice text, and session descriptions adjust to reflect the correct product name.

Export & Integration

Structured Export Formats

Six export formats, all generating plain text optimized for copy-paste into support systems:

  • Quick Summary — device info, top accounts with live balances, top errors with severity, portfolio total. Formatted for support ticket notes.
  • Full Export — complete device info, all accounts with balances and fiat values, all errors with severity and recommended actions, APDU count, data quality score, session activity summary. For escalation to Tier 2 or engineering.
  • Customer Summary — account-focused with live balances, fiat values, and portfolio total. For customer communication. Available in both Diagnostic and Customer View modes.
  • Error Export — all errors with severity, diagnostic title, recommended action, category, entry reference, timestamp, and common causes. Optimized for error-focused ticket investigation.
  • Filtered Timeline Export — current filtered and searched timeline entries as JSON. Preserves active filter state.
  • Account Data Export — all accounts with chain identifier, address, derivation mode, and operation count as JSON.

All copy formats include live blockchain balances and fiat values when available. Formats that include account data add a ⚠ NO SYNC DATA warning line when the log contains no sync information, so recipients are aware of the data source limitations.

Data Sovereignty

All log parsing, analysis, pattern matching, and data correlation runs entirely in the browser via client-side JavaScript. No log data is transmitted to any server under any circumstance.

External contacts made during analysis (addresses only, no log content):

  • Public blockchain APIs — account addresses are sent to query on-chain balances. No log content, error data, or metadata is included.
  • CoinGecko API — chain currency identifiers (e.g., ethereum, bitcoin) are sent to retrieve USD prices. No account or user data is included.

No cookies are set. No data is written to localStorage or sessionStorage. No analytics or telemetry of any kind. The tool can operate fully offline except for live balance fetching and fiat pricing — all other features function without network access.

Log files may contain sensitive customer data including account addresses, device identifiers, and user IDs. Handle exported summaries and raw log data according to your organization's data handling and privacy policies.
